## 2. Findings

### 2.1 Authorization model is binary

**Finding:** [ ] Pass [x] Fail [ ] N/A

| Severity | Issue | Location | Impact |
|---|---|---|---|
| High | All authenticated endpoints are protected by `anyRequest().authenticated()` only — no role-based access control. `JwtAuthFilter.authenticate(...)` builds a `UsernamePasswordAuthenticationToken(claims.getSubject(), null, List.of())` with **empty authorities**. No `@PreAuthorize`, no `hasRole(...)` calls anywhere in `web/`. | `backend/src/main/java/za/co/csnx/demo/security/JwtAuthFilter.java:71` & `backend/src/main/java/za/co/csnx/demo/config/SecurityConfig.java:58` | A logged-in user can call every authenticated endpoint, including any future admin / metadata-mutation endpoint added later. RBAC must be added before the surface grows. |

### 2.2 CORS is permissive for credentialed requests

| Severity | Issue | Location | Impact |
|---|---|---|---|
| Medium | `CorsConfiguration.setAllowedHeaders(List.of("*"))` combined with `setAllowCredentials(true)` — modern browsers accept this combination (the wildcard expands to the request's `Access-Control-Request-Headers` list), but the policy permits arbitrary headers from any allowed origin. | `backend/src/main/java/za/co/csnx/demo/config/SecurityConfig.java:67-68` | Limited — origins are still constrained by `allowedOrigins`. But any future XHR pre-flight from an allowed origin can carry arbitrary headers without server-side review. |
| Low | Allowed origins are hardcoded to `http://localhost:5173` and `http://localhost:8081` in `application.yml:54-56`; no separate prod profile sets them. | `backend/src/main/resources/application.yml:54-56`; `backend/src/main/resources/application-prod.yml:9-12` (prod overrides only `8081`) | Prod deployments must remember to set `APP_SECURITY_CORS_ALLOWEDORIGINS` env var or production CORS rejects all browser callers. |

### 2.3 CSRF disabled with cookie auth in play

| Severity | Issue | Location | Impact |
|---|---|---|---|
| High | `http.csrf(AbstractHttpConfigurer::disable)`. Acceptable for a pure bearer-only API, but `/api/auth/refresh` reads the `refreshToken` cookie — that endpoint is reachable cross-origin via a forged POST from any allowed origin. With `SameSite` also unset on the cookie (see `authentication.md` finding 3.1), the CSRF surface is real. | `backend/src/main/java/za/co/csnx/demo/config/SecurityConfig.java:42` | An attacker controlling an allowed-origin asset can trigger refresh-token rotation against a victim's session. |

### 2.4 Endpoint advertised in docs but not exposed

| Severity | Issue | Location | Impact |
|---|---|---|---|
| Medium | `README.md:96` documents `POST /api/auth/register` as a public endpoint, but `AuthController.java` has no `register` handler (only `login`, `refresh`, `logout`). Either the README is stale, or the endpoint moved without README update, or the seeded `wcs` user replaces the registration flow. Confusing surface for security review. | `README.md:96` vs. `backend/src/main/java/za/co/csnx/demo/web/AuthController.java` | Low direct impact, but a divergence between docs and reality is a signal that other endpoint metadata may be wrong. |
| Medium | `AuthController.java` lacks a `change-password` handler, yet `frontend/src/app/auth/AuthProvider.tsx:254` POSTs to `/api/auth/change-password`. Either the endpoint lives in a controller not read during this audit, or the frontend talks to a dead URL. | Cross-reference between `AuthProvider.tsx:254` and the `web/` package | Medium — silent 404 / 405 may mask a missing security control. |

### 2.5 Detailed error responses

| Severity | Issue | Location | Impact |

| Medium | `server.error.include-message: always` and `include-binding-errors: always` cause Spring's default error pages to surface exception messages and binding errors to the client. The `GlobalExceptionHandler` returns RFC 7807 `ProblemDetail` with `detail` populated from raw exception messages (e.g. `ex.getMessage()` at lines 79, 100). | `backend/src/main/resources/application.yml:25-27`; `backend/src/main/java/za/co/csnx/demo/common/GlobalExceptionHandler.java:79, 100` | Internal exception text leaks into client responses — useful for debugging, but reveals stack-internal structure. ## 3. Recommendations

### Immediate
- Disable Swagger UI in prod (`@Profile("!prod")` on `OpenApiConfig`, or `permitAll → authenticated` in a prod-only `SecurityFilterChain`).
- Reconcile `README.md:96` with `AuthController.java` — either restore `/api/auth/register` or remove from the docs.

### Short-term
- Add role-based authorities to the JWT (claim `roles` or `authorities`), propagate into `JwtAuthFilter.authenticate(...)`, and replace `anyRequest().authenticated()` with method-level `@PreAuthorize`.
- Constrain `setAllowedHeaders` to an explicit list (`Authorization`, `Content-Type`, `X-Requested-With`).
- Add `bucket4j` rate-limiter on `/api/auth/**` and `/api/lookup/init`.
- Add CSRF protection on cookie-authenticated endpoints (or treat the refresh cookie as Bearer via a header-side token).

### Long-term
- Generate a typed OpenAPI client for the frontend so the contract is enforced at compile time, eliminating dead URLs like the suspected `/api/auth/change-password`.
- Per-endpoint observability (`Micrometer` + per-route timers) so brute-force / scraping attempts surface in metrics.

---

## Conclusion

The API surface is small and well-structured (RFC 7807 errors, OpenAPI doc, JWT bearer). The two High-severity gaps — empty authorities on the security context and CSRF disabled while a cookie is in play — are both quick fixes that materially change the security posture.
